# GATE 5: FISCAL ARCHITECTURE (THE OBSERVER)
> **Principle:** The "Eye of Sauron" approach. The Fiscal Module sees everything, but touches nothing.

---

## 1. THE PROBLEM
Traditional POS systems mix "business logic" with "tax logic".
*   *Example:* "If I emit a burger order, I must calculate ICMS tax inside the order class."
*   *Risk:* Changing a tax rule might break the kitchen printer. Or a government API downtime halts the restaurant service.

## 2. THE SOLUTION: OBSERVER PATTERN
We treat the Sealed Core (Gates 0-4) as a **Read-Only Source of Truth**.

### Data Flow
1.  **Core Transaction:** `PAYMENT_CONFIRMED` + `PAYMENT_SEALED` are committed to Postgres. (Atomic)
2.  **Notification:** The DB (via Listen/Notify) or a Polling Service wakes up the Fiscal Adapter.
3.  **Transformation:** The Fiscal Adapter maps `CoreEvent` -> `TaxDocument` (e.g., XML for SAT).
4.  **Transmission:** The Adapter sends data to the Government.
5.  **Receipt:** The Adapter stores the `TaxReceipt` in a **separate** schema/table (`fiscal_records`).

### Separation of Concerns
| Component | Responsibility | Access |
|-----------|----------------|--------|
| **Core** | Make Money | Write: `event_store` |
| **Legal** | Prove Fact | Write: `legal_seals` |
| **Fiscal** | Report Fact | Read: `event_store`, Write: `fiscal_event_store` |

## 5. FISCAL PERSISTENCE (GATE 5.1)
To ensure traceability of **what was reported**, we maintain a dedicated `fiscal_event_store`.
*   **Purpose:** Prove to an auditor that we *did* send the XML to the government.
*   **Linkage:** Each record points back to a `LegalSeal`.
*   **Content:** Contains the exact XML sent and the Protocol received.

---

## 3. THE INTERFACE

```typescript
interface FiscalObserver {
  // Named "onSealed" to emphasize it waits for the seal
  onSealed(seal: LegalSeal, event: CoreEvent): Promise<FiscalResult>;
}

type FiscalResult = {
  status: "SUCCESS" | "REJECTED" | "QUEUED";
  gov_protocol?: string; 
  error_details?: string;
}
```

## 4. FAILURE HANDLING (ASYNC)
If the Government API is down:
1.  **Core:** Continues operating (selling).
2.  **Fiscal:** Enters "Offline/Contingency Mode".
3.  **Queue:** Unreported events accumulate in a persistent queue.
4.  **Retry:** Worker retries transmission later.

**Critical:** The "Sale" happened. The "Tax" is an obligation generated by the sale. The technical failure to transmit XML does not invalidate the reality of the sale (Financial Reality precedes Bureaucratic Reality).
