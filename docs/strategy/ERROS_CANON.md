# ERROS_CANON — Catálogo de falhas inevitáveis e Radar de Falhas

Não prever features; prever forças. Criar defesas antes de chegarem.

As 7 forças inevitáveis em sistemas como o ChefIApp OS: multiplicidade de canais; concorrência e estado partilhado; dinheiro vira auditoria; menu vira constituição; offline/latência/reconexão; observabilidade e responsabilização; escopo político.

---

## Radar de Falhas (3 camadas)

### Camada A — Invariantes (leis que nunca podem quebrar)

| Invariante                             | Onde está garantido                                                                                                             |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| Served ≠ Paid                          | [CORE_FINANCE_CONTRACT_v1.md](../contracts/CORE_FINANCE_CONTRACT_v1.md)                                                         |
| KDS não esconde status desconhecido    | [ORDER_STATUS_CONTRACT_v1.md](../contracts/ORDER_STATUS_CONTRACT_v1.md), [CHECKLIST_KDS_FLUXO.md](./CHECKLIST_KDS_FLUXO.md)     |
| Pedido congela snapshot do menu        | [MENU_BUILDING_CONTRACT_v1.md](../contracts/MENU_BUILDING_CONTRACT_v1.md)                                                       |
| Só RPC muda status (Core é autoridade) | [ORDER_STATUS_CONTRACT_v1.md](../contracts/ORDER_STATUS_CONTRACT_v1.md); CoreOrdersApi create_order_atomic, update_order_status |

**Regra:** Toda feature nova deve declarar 1–3 invariantes. Se não forem escritos, viram bugs fantasmas.

---

### Camada B — Pontos de fratura

Sítios onde o sistema quebra com facilidade:

- Mudança de status (transições, duplo clique, idempotência)
- Criação e fecho de pedido (constraint uma mesa = um OPEN, Core Finance)
- Pagamentos e estornos (autorização vs captura, refund, reconciliação)
- Sync offline (reconexão, duplicados, conflitos)
- Integrações externas (payload inválido, status fora do enum)

**Validação:** Testes de caos e replay nesses pontos. [CHECKLIST_OPERACIONAL_TPV_KDS_CLIENTE.md](./CHECKLIST_OPERACIONAL_TPV_KDS_CLIENTE.md), [CHECKLIST_MUNDO_ESTRESSADO.md](./CHECKLIST_MUNDO_ESTRESSADO.md), world-chaos ([CLI_CHEFIAPP_OS.md](./CLI_CHEFIAPP_OS.md)).

---

### Camada C — Trilha de eventos

Sem order_events / financial_events (ou equivalente) não é possível: replay, auditoria, simulação, explicação de bug.

**Referência:** [EVENTS_AND_STREAMS.md](../contracts/EVENTS_AND_STREAMS.md). O schema actual pode ainda não expor tabelas mínimas de eventos; EVENTS_MINIMAL v1 é o próximo passo recomendado para memória e cognição operacional.

## Quando uma pergunta nova é válida ou ruído

Quando alguém trouxer uma pergunta nova no futuro, aplicar três checks. Isso protege o sistema do crescimento inconsciente, não do crescimento em si: ou a pergunta vira E## novo, ou é respondida pelo documento, ou é descartada sem drama, ou activa ritual de mudança.

**CHECK 1 — Ela viola algum invariante?**
Se sim → alarme máximo. (Ver Camada A do Radar de Falhas; [ORDER_STATUS_CONTRACT_v1](../contracts/ORDER_STATUS_CONTRACT_v1.md), [CORE_FINANCE_CONTRACT_v1](../contracts/CORE_FINANCE_CONTRACT_v1.md), [MENU_BUILDING_CONTRACT_v1](../contracts/MENU_BUILDING_CONTRACT_v1.md).)

**CHECK 2 — Ela já está coberta por algum E##?**
Se sim → responder com o documento, não com opinião. (Usar este catálogo e a tabela Perguntas inevitáveis.)

**CHECK 3 — Ela cria um novo tipo de falha?**
Se sim → novo E## entra no ERROS_CANON (com Sintoma, Descrição, Como detectar, Como corrigir, Referência).

Se não passar por nenhum dos três → provavelmente é ruído, feature creep ou política. Não aceitar sem ritual de mudança ou justificação explícita.

### Três zonas onde o sistema ainda vai quebrar (mesmo com este catálogo)

**A) Ambiguidade semântica entre humanos.** Duas áreas usam a mesma palavra para coisas diferentes (ex.: "pronto" para cozinha vs "entregue" para salão; "fechar pedido" operacional vs financeiro; "cancelar" vs "anular" fiscal). Defesas parciais: ORDER_STATUS_CONTRACT, Served ≠ Paid, Core Finance separado. A falha seria aceitar status ou regra nova (ex.: INTERMEDIATE, EM_ENTREGA) sem ritual — já existe lei para isso ([LEI_EXISTENCIAL_CHEFIAPP_OS.md](./LEI_EXISTENCIAL_CHEFIAPP_OS.md), ritual de mudança).

**B) Excepções humanas "emocionais".** Excepção vira hábito e não deixa trilha (desconto "por favor", fechar "depois", "resolve amanhã"). Já mapeado: E30 (excepção financeira sem doc), governança esquecida, ritual de mudança, zonas intocáveis. A maioria dos sistemas morre aqui; este tem anticorpos.

**C) Tempo longo (meses, não dias).** Quebra em relatórios históricos, regras que mudaram sem versionamento, contratos não versionados. Já antecipado: snapshot de menu, contratos v1, ritual de mudança, ERROS_CANON como memória institucional. Quando chegar: versionamento explícito de contratos (v2, v3) — não obrigatório agora.

---

**Frase-lei:** O sistema vivo não elimina perguntas. Elimina o pânico quando elas aparecem.
