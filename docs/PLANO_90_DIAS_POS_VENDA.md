# Plano 90 Dias Pós-Venda — v1 → v2

**Data:** 2026-01-26  
**Status:** 🎯 PLANEJAMENTO

**Objetivo:** Transformar feedback de early adopters em v2 baseado em uso real.

---

## 🎯 Princípio-Base

> **"v2 evolui baseado em uso real, não suposições."**

Não planejamos v2 no vácuo. Planejamos baseado em:
- Uso real (3-5 restaurantes)
- Feedback real
- Dados reais

---

## 📊 Estrutura (90 Dias)

### Dias 1-30: Observação
- Coletar dados diariamente
- Anotar comportamentos inesperados
- Identificar padrões
- Documentar "como funciona na prática"

### Dias 31-60: Análise
- Analisar padrões identificados
- Priorizar ajustes necessários
- Decidir: ajustes cirúrgicos ou features v2
- Validar que ajustes funcionam

### Dias 61-90: Evolução
- Implementar ajustes prioritários
- Validar com usuários
- Preparar v2 baseado em dados reais
- Decidir: continuar v1 ou lançar v2

---

## 📋 Plano Detalhado

### DIAS 1-30: OBSERVAÇÃO

#### Semana 1: Baseline
**Objetivo:** Estabelecer baseline de uso

**Métricas a coletar:**
- Pedidos criados/dia
- Tarefas geradas/dia
- Tarefas resolvidas/dia
- Itens na lista de compras
- Tempo médio de preparo (real vs. estimado)
- Taxa de erro (pedidos com problema)

**Feedback a coletar:**
- O que funciona bem?
- O que não funciona?
- O que falta?
- O que sobra?

**Ações:**
- [ ] Coletar métricas diariamente
- [ ] Anotar comportamentos inesperados
- [ ] Coletar feedback semanal
- [ ] Documentar "como funciona na prática"

- Qual é o tempo real vs. estimado?

**Ações:**

### DIAS 31-60: ANÁLISE

#### Semana 5-6: Análise Profunda
**Objetivo:** Analisar padrões e priorizar

**Análise a fazer:**
- Quais problemas são críticos?
- Quais melhorias são prioritárias?
- O que pode ser ajuste cirúrgico?
- O que precisa ser feature v2?

**Priorização:**
- **Crítico:** Bloqueia uso → Ajuste imediato
- **Alto:** Impacta uso → Ajuste rápido
- **Médio:** Melhora uso → Feature v2
- **Baixo:** Nice to have → Feature v2+

**Ações:**
- [ ] Analisar todos os padrões
- [ ] Priorizar problemas
- [ ] Priorizar melhorias
- [ ] Decidir: ajustes vs. features

## 🚫 O Que NÃO Fazer (90 Dias)

### ❌ Adicionar Features Prematuramente
- Não adicionar features sem demanda real
- Não adicionar features "por precaução"
- Não adicionar features genéricas

**Razão:** v2 evolui baseado em uso real, não suposições.

### ❌ Ignorar Feedback
- Não ignorar problemas críticos
- Não ignorar padrões claros
- Não ignorar demanda real

**Razão:** Feedback real é ouro. Não desperdiçar.

### ❌ Refatorar "Por Precaução"
- Não refatorar código que funciona
- Não otimizar prematuramente
- Não mudar arquitetura sem necessidade

**Razão:** Código que funciona em produção > código "perfeito".
