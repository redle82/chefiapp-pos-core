ChefIApp™ — O Sistema Operacional Vivo do Restaurante

Uma aplicação única que une disciplina (tarefas) + operação (TPV) + estado do sistema (dashboard) + pessoas (staff).

O ChefIApp nasceu do espírito das tarefas: transformar caos em ritmo, equipa em máquina humana coordenada, e operação em clareza.

Visão e Alma

O Espírito: Task System

O sistema de tarefas é o núcleo simbólico e funcional do ChefIApp. Ele é o que um restaurante sempre precisou e quase nunca teve:

  • Rotina clara em tempo real
  • Execução com cronómetro
  • Responsabilidade por papel
  • Pontos / XP / streak (quando apropriado)
  • Notificações internas
  • Feedback operacional sem gritaria
  • Disciplina que não depende do "chefe estar em cima"

A tarefa não é checklist. É pulso. Ela coordena:

  • preparo (cozinha)
  • atendimento (salão)
  • limpeza (turnos e padrões)
  • abertura/fecho (processos)
  • consistência (qualidade)

Camadas do Sistema

A mesma alma, em quatro corpos:

1) Configuração (Backoffice silencioso)

Configura o restaurante em qualquer ordem. Sem wizard. Sem onboarding teatral.

2) Dashboard (visão de estado)

Mostra: se está pronto para vender; o que falta; o que está acontecendo hoje.

3) Operação (TPV / KDS / Turnos)

Executa: abrir turno; vender; fechar turno; fluxo de pedidos.

4) Core (soberano)

Valida e consolida (financeiro, auditoria, registros). O staff pode rodar mesmo sem o core em demo/pilot, com runtime operacional local.

Papéis

Dono (Owner)

  • visão do sistema
  • saúde operacional
  • relatórios e decisões
  • controle de permissões
  • habilitar modos (demo/pilot/prod)

Gerente (Manager)

  • execução diária
  • turnos
  • ajustes de cardápio e preços
  • acompanhamento de equipa
  • resolução de alertas operacionais

Staff (Waiter / Kitchen / Cleaning)

  • tarefas
  • presença / local (location)
  • fluxo operacional do papel
  • execução rápida e objetiva

Rotas e Apps

Portal (web)

  • /dashboard (centro)
  • /config/* (identidade/local/cardápio/publicação)
  • /tpv (operação real)
  • /tpv/preview (simulação)
  • /system/* (saúde, alertas, estado técnico)
  • /setup/restaurant-minimal (único setup mínimo, sem wizard)

AppStaff (operacional humano)

  • /app/staff?role=manager|waiter|kitchen|cleaning
  • funciona em demo/pilot sem core, com runtime operacional local: locationsStore, createLocalContract, estado de papel por aba

Módulos do Produto (o corpo inteiro)

Núcleo (MVP vivo)

  • Task System (alma) — tarefas por papel; cronómetro; status (pendente/em curso/concluída); pontos/XP (quando habilitado)
  • Operação — Turnos & Caixa; TPV (carrinho, pagamento, envio de pedido); pedidos em tempo real
  • Config — identidade/local/moeda; cardápio e publicação
  • Dashboard — "Operação: pronto/incompleto"; "Operacional (hoje)"; "Turno atual"; "Alertas"

Satélites (Fase 2/3, sem poluir o MVP)

  • Fidelização / missões externas (Customer Loop satélite)
  • Integrações avançadas
  • Relatórios profundos
  • Automação por IA (sugestões de "o que fazer agora")

Fluxos-Chave

Telefone → Dashboard (sem onboarding)

  1. SMS login
  2. se não há restaurant_id → /setup/restaurant-minimal (nome/país/moeda/tipo)
  3. cai no /dashboard e o estado manda

Dashboard → TPV (ações bloqueiam, navegação não)

  • se configuração incompleta → mensagem + botão "Ir para configuração"
  • se turno fechado → "Abrir turno"
  • venda só ocorre com turno aberto

Staff (aba separada)

  • escolhe role via query param
  • escolhe location (ou "Nenhuma localização ativa")
  • executa tarefas e fluxo do papel

Offline e Resiliência (o que torna "especial")

O ChefIApp não é frágil. Ele tolera o mundo real: queda de internet; core offline em demo/pilot; staff continua com runtime operacional local; sincroniza quando voltar.

O Core valida. O App executa. A operação não para.

O que o app tem (terminologia precisa)

O app tem um runtime operacional local com persistência temporária:

  • mini-estado local
  • armazenamento local (localStorage / IndexedDB)
  • contrato operacional local
  • fila de ações; estado temporário; capacidade de continuar a operar

Não tem: Core soberano; livro-razão final; fecho oficial do turno; validação financeira final. Tem: runtime operacional local, fila de ações, estado temporário.

Fluxo real

Online: App (Staff/TPV) → ações → Core (valida + grava) → Dashboard (lê estado). Tudo em tempo real.

Offline: App guarda ações localmente; mantém estado de turno local; permite pedidos, cozinha, limpeza; marca tudo como "pending". Nada trava.

Quando a conexão volta: App envia fila → Core valida/consolida → Dashboard reflete estado final. Se alguma ação for inválida: o Core recusa; o Dashboard alerta; o sistema não quebra.

É único no contrato, não no runtime. Contrato único: tipos de ação, regras de validação, permissões. Runtimes múltiplos: browser do garçom, tablet da cozinha, desktop do dono. Todos obedecem as mesmas regras, mas não dependem do mesmo pulso.

Checklist final

  • O app funciona offline
  • Tem armazenamento local
  • Aguenta horas de trabalho
  • Tem Core próprio
  • Sincroniza depois
  • Core continua soberano

Roadmap em Fases (sem quebrar o MVP)

  • Fase 1: disciplina + operação mínima real (tarefas + TPV + turnos)
  • Fase 2: TPV mais profundo + integrações + relatórios
  • Fase 3: IA operacional + loops externos (satélites)
