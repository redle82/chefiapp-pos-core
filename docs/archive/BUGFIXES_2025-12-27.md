## 🐛 Bug 1: CLOSED State Terminal Flag Contradiction ✅ **FIXED**

### Problem
The `CLOSED` state in `session.state-machine.json` had a `RESET` transition to `INACTIVE`, but:
1. The state machine executor checks `terminal` flag **before** checking transitions
2. The documentation (`01_CORE_STATES.md`) states `CLOSED` is terminal and cannot transition
3. This made the `RESET` transition dead code that could never execute

### Root Cause
- JSON had transition but no `terminal: true` flag
- Documentation contradicted JSON definition
- Executor logic: `if (stateDef.terminal) return error` runs before transition check

### Fix Applied
1. **Removed `RESET` transition** from `CLOSED` state in `session.state-machine.json`
2. **Added `terminal: true`** flag to `CLOSED` state
3. **Removed `RESET` event** from events list
4. **Updated documentation** to remove contradictory `CLOSED → INACTIVE` transition

### Files Changed
- `state-machines/session.state-machine.json`
- `:blueprint/01_CORE_STATES.md`

### Result
✅ State machine is now consistent: `CLOSED` is terminal, no transitions allowed

## 🐛 Bug 2: rebuildState Discarding Return Value ✅ **VERIFIED CORRECT**

### Problem Reported
The `rebuildState` function was reported to declare `state` with `const` and call `applyEvent(state, event)` but discard the return value, breaking event sourcing replay.

### Investigation
**Code Analysis (lines 234-248):**
```typescript
export function rebuildState(events: CoreEvent[]): ProjectionState {
  let state: ProjectionState = {  // ✅ Uses `let`, not `const`
    sessions: new Map(),
    orders: new Map(),
    orderItems: new Map(),
    payments: new Map(),
  };

  // Apply events in order (deterministic)
  // CRITICAL: applyEvent returns NEW state (immutable), must use return value
  for (const event of events) {
    state = applyEvent(state, event);  // ✅ Correctly assigns return value
  }

  return state;
}
```

### Verification
- ✅ **Checked current code**: Uses `let`, assigns return value correctly
- ✅ **Checked git history**: Same correct implementation in HEAD and main
- ✅ **Checked for other implementations**: Only one `rebuildState` function exists

### Result
✅ **Code is correct** - The function properly:
- Uses `let` (not `const`) for `state`
- Assigns return value: `state = applyEvent(state, event)`
- Returns the final state
- Comment explicitly documents the critical requirement

**Note:** This bug may have existed in a previous version but has been fixed. Current code is correct and verified.

## 🐛 Bug 3: withLock Race Condition ✅ **FIXED**

### Problem
The `withLock` function had a race condition:
1. Multiple callers (B and C) wait on same lock (A)
2. When A releases, both B and C resume simultaneously
3. After `await existingLock`, code proceeds without re-checking
4. Both B and C can create new locks, allowing concurrent execution

### Root Cause
```typescript
// ❌ WRONG: Race condition
const existingLock = this.locks.get(entityId);
if (existingLock) {
  await existingLock;  // Both B and C resume here
}
// No re-check - both proceed to create lock
this.locks.set(entityId, lock);
```

### Fix Applied
```typescript
// ✅ CORRECT: Loop until lock acquired atomically
while (true) {
  const existingLock = this.locks.get(entityId);
  if (!existingLock) {
    break;  // No lock exists, we can acquire it
  }
  await existingLock;  // Wait for release
  // Loop continues - re-check if another waiter acquired it
}
// Only one waiter reaches here at a time
this.locks.set(entityId, lock);
```

### Files Changed
- `core-engine/repo/InMemoryRepo.ts`

### Result
✅ Lock acquisition is now atomic - only one waiter can acquire lock at a time
