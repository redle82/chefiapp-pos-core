# Auditoria de Violações ao Financial Core

**Propósito:** Documentar usos de Supabase, Firebase ou lógica financeira no cliente que violam ou contornam o [CORE_FINANCIAL_SOVEREIGNTY_CONTRACT.md](./CORE_FINANCIAL_SOVEREIGNTY_CONTRACT.md). O Docker Core é o único Financial Core; Supabase/Firebase não podem ser fonte de verdade.

---

## Declaração P0 (estado atual)

**Nenhuma escrita/leitura financeira passa por Supabase em modo Docker.** (Orders, Order Items, Fiscal Event Store, Inventory, Billing.) Domínio crítico (gm*orders, gm_order_items, fiscal_event_store, inventory*\*, billing) é exclusivamente Docker Core. Supabase permanece apenas para Auth temporário e serviços lab/legado fora do caminho crítico (bloqueados pelo shim quando backend = Docker).

**P0 RESOLVIDO.** Data: 2026-02-03. Commit aproximado: ramo `core/frozen-v1` / remediação Orders, Stock, Fiscal, Billing concluída.

**Classificação:**

- **a) Legacy** — Código antigo, scripts ou testes; não usado como Core em runtime normal; a migrar ou a marcar como não autoritário.
- **b) Technical debt** — Uso actual que deve evoluir para Core API ou ser explicitamente marcado como integração subordinada (não autoritativa).
- **c) Architectural violation** — Uso de BaaS ou cliente como fonte de verdade para estado financeiro, pedidos, totais, pagamentos ou SLA; violação directa do contrato.

**Regras a fazer cumprir:**

- Nenhum terminal calcula totais, preços, estado de pagamento ou SLA localmente (Core calcula e expõe).
- Nenhuma UI persiste verdade financeira ou operacional (Core persiste).
- Nenhum contrato omite referência ao Financial Core (todos têm secção Sovereignty).
